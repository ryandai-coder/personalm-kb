## 0.1 验证关卡(在写一行代码前必须先过)

> 整个项目能不能做出"AI 像密友一样懂我"的体验,**取决于 prompt 工程而非系统工程**。系统再精巧,如果冷启动产出的自我画像无法让人满意,就是无效投入。所以在 M1 之前先过两个低成本关卡。

### 关卡 A:yourself-skill 试跑(30 分钟,本周做)
- 把现有的微信收藏/备忘录里关于自己的描述、近期一两段对话凑成原料,跑一次开源项目 [`notdog1998/yourself-skill`](https://github.com/notdog1998/yourself-skill)。
- **判定标准**:产出的 Self Memory 和行为翻译是否让我读完有"是的,这就是我"的感觉?是否能给到 Self Memory 5 维度的 v0 草稿水准?
- **不通过的处置**:这个项目不必继续。说明 LLM 现阶段对"自我画像"的捕捉还不到我期待的密友级。
- **通过的处置**:其产出的 Self Memory 文件作为 L1 基底语料的种子,行为翻译规则作为 §3.4 翻译表的初始输入。

### 关卡 B:Claude Projects 手工对话试用(2 周)
- 不写任何代码,直接在 Claude Projects 上传关卡 A 的产物 + 几条情绪日记 + 几个核心人物档案,日常用网页/手机 Claude App 对话两周。
- **判定标准**:有没有出现"它给的反馈比通用 ChatGPT 显著更懂我"的瞬间?有几次?
- **不通过的处置**:重新审视 §1.2 价值主张是否真实,可能需要把"密友"降级为"结构化的私人 RAG"。
- **通过的处置**:进入 M1 工程化。

> **核心原则**:工程化是为了把已经验证可行的对话体验**自动化和长期化**,不是为了把不存在的体验**召唤出来**。

## 0.2 项目策略与"我做什么、不做什么"

### 不从零写
- **RAG/Vault 后端用 Khoj**:已经有 Markdown ingest、Obsidian 同步、多模型路由、Web/Desktop UI。社区维护到 2026 仍活跃。
- **冷启动蒸馏借鉴 yourself-skill 方法论**:不只是"跑一次拿草稿"，而是系统性借用其三个核心设计：(1) 材料蒸馏优于访谈式自我陈述；(2) 标签→行为规则翻译层；(3) Self Memory 5 维度统一 schema。详见 §2.4、§5.3.5、§5.8、§7。
- **不试图取代 Obsidian / Khoj / Claude**:数据是 Markdown,任何环节都可平迁。

### 自己写的部分(差异化价值所在)
1. **Telegram Bot 主入口** + 转写 + 路由(Khoj 没有 Telegram 集成)。
2. **编排层**:分类归档、Persona 切换、决策模板、追问/复盘调度。
3. **三层蒸馏模型**(见 §5.8):yourself-skill 只有一次性蒸馏(Layer A),本项目增加了持续蒸馏(Layer B,每次对话提取行为证据)和周期性模式检测(Layer C,周/月/季复盘发现矛盾和新模式)。这是"越用越懂我"的工程实现。
4. **Correction 闭环** + 证据强度分级(见 §5.3.1.1、§5.4.2):yourself-skill 有纠正机制但无证据分级。本项目增加四级证据强度 + 冲突标注，让 AI 知道什么确定、什么猜测、什么已否定。
5. **审计层** + `/undo` + `/correct`(Khoj 不提供)。

### 跨项目对照(避免混淆)
| 项目 | 它做什么 | 我用它做什么 |
|---|---|---|
| Khoj | RAG over notes,Obsidian 集成 | 当 RAG/Vault 引擎,API 调用 |
| yourself-skill | 一次性蒸馏"数字分身"，产出 Self Memory + Persona + 行为翻译表 | **方法论借鉴**：材料蒸馏、标签翻译、Self Memory schema、Correction 闭环。**不**用其"数字分身"运行规则和 Layer 2 说话风格模仿 |
| Obsidian | Markdown 编辑器 + Vault | 人工编辑入口、深度复盘界面 |
| Claude Projects | 验证阶段对话承载 | 关卡 B 用,M1 后退出 |
| 我自己写的 | Telegram + 编排 + 三层蒸馏 + 证据分级 + 审计 | 项目主体,差异化所在 |

> **核心定位差别**:yourself-skill 的目标是"造分身让 AI 像我说话"；本项目的目标是"造陪伴者让 AI 和我说话"。借它的数据模型（怎么理解用户），不借它的运行规则（怎么模仿用户）。

## 1. 项目目标(Why)

### 1.1 一句话目标
建立一个**长期累积、本地可控、对话式输入、AI 智能输出**的个人知识库,让 AI 能像了解我多年的密友一样,在分析、判断、决策与心理疏导上持续帮助我。

### 1.2 核心价值主张
1. **越用越懂我**:不像通用 ChatGPT 每次冷启动,本系统沉淀我的过去/现在/价值观,回答带"个人上下文"。
2. **降低记录摩擦**:对话即记录,AI 自动归类、结构化、关联,人不必维护目录。
3. **隐私可控**:Markdown 原文永不出设备；语料经脱敏后可选云端模型做 embedding 和推理（详见 §6.3.3）。
4. **多入口、低打扰**:Telegram 即时记录,Obsidian 深度复盘,手机/电脑/路上都能用。
5. **支持反思与决策**:不是被动笔记本,是主动陪练——会追问、会提醒、会做决策模板。

### 1.3 非目标(本项目不做)
- 不做团队协作、知识共享。
- 不做通用搜索引擎/Wiki 替代品。
- 不做投资量化、健康医疗诊断等需要专业数据源的场景。
- 不做"AI 替我决策",AI 永远是建议者,人是决策者。

### 2.2 各分类要素清单(摘要)

> 完整模板字段在 §5.3 数据模型给出。

- **§1 亲情**:成员档案(年龄/健康/职业/性格/忌讳)、关系动态、未解心结、家族财务、纪念日。
- **§2 爱情**:伴侣画像、关系阶段、共同目标、价值观差异、冲突模式、浪漫记忆库。
- **§3 友情**:核心朋友(<10)、弱连接、人情账本、想加深/淡出的关系。
- **§4 职场关系**:上司/同级/下属/跨部门关键人/外部人脉。
- **§5 当前岗位**:公司/团队/绩效/KPI/薪酬/政治/不爽点。
- **§6 技术栈**:已掌握/在学/想学;作品集;盲区清单。
- **§7 职业规划**:5/10 年目标;副业;跳槽窗口判定。
- **§8 健康**:体检/慢性病/家族病史;作息;睡眠-情绪关联。
- **§9 情绪日记**:事件→感受→想法→行为(CBT 框架);触发器;自动想法。
- **§10 价值观**:个人原则;偶像;近期被颠覆的观念。
- **§11 兴趣**:长期爱好/近期沉迷/想体验的事/影音书品味。
- **§12 财务**:收入结构/储蓄/负债/投资/保险/家庭账本/大额计划。
- **§13 时间精力**:周分配/高效低效时段/复利型 vs 消耗型。
- **§14 决策档案**:背景/选项/约束/不确定性/决定/事后复盘。
- **§15 复盘**:周/月/季/年自由复盘;失败清单。
- **§16 灵感**:对话金句/读到的好段落/可能升级的想法。
- **§17 阅读学习**:书/文章/播客/课程笔记。
- **§18 通讯历史**:重要微信对话/邮件/Slack 导出,作为关系档案与情绪事件的事实底料(详见 §6.6)。
- **§19 人生时间线**:出生→现在的关键节点;童年/原生家庭/求学/工作/关系/转折。
- **§20 模式**:反复出现的处境;"剧本";防御机制;依恋类型。

### 2.4 标签→行为规则翻译层

> 借鉴 yourself-skill 的"标签翻译表"方法论：用户用抽象标签描述自己（"社恐""纠结体"），但这些标签对 AI 而言不可执行。需要将每个标签翻译为具体的行为规则，AI 才能正确理解用户的实际表现。

不是所有 20 个分类都需要翻译层。需要翻译的是**涉及自我认知的分类**（§8-§13、§19-§20），以及**人物关系分类**（§1-§4）。

#### 核心自我标签翻译示例

| 用户说的 | 翻译为 AI 可用的行为规则 |
|----------|------------------------|
| "我和妈妈关系复杂" | 不是单纯的亲近或疏离。在不同话题上分歧很大（春节回不回家、职业选择），但底层在意对方。不要在涉及她婚姻的话题上深入，也不要在她面前显得"翅膀硬了"。 |
| "我容易焦虑" | 遇到不确定性时倾向反复确认、列计划来缓解；不是恐惧型焦虑，更多是控制感缺失。会表现为「行动狂躁」—突然列很多计划、查很多资料。 |
| "我对现在的工作不满意" | 不满意的维度需要用行为判断：是薪资/P0 成长/P0 意义感/P0 人际关系？不同维度的不满意对应完全不同的建议方向。需要追问而非假设。 |
| "我拖延" | 不是懒，是对"做不好"的恐惧导致的回避。通常在 deadline 前突然高效。不要用"你应该早点开始"这种无效建议。 |
| "我想得多做得少" | 内心有很强的行动标准但启动困难，思虑过多消耗了行动能量。适合用"下一步最小行动是什么"来破冰，而非宏大计划。 |
| "我嘴硬心软" | 嘴上说"随便""无所谓"，行动上会偷偷做好。不先道歉但会用行动示好。AI 需要注意识别：他说的"算了"不一定真的算了。 |

#### 人物关系标签翻译示例

| 关系分类 | 用户可能说的 | 翻译为 AI 可用的规则 |
|----------|------------|-------------------|
| §1 亲情 | "我爸不太说话" | 不是冷漠，是表达方式不同。他的关心体现在行动（给你寄东西、修东西）而非语言。不要把他的沉默解读为不关心。 |
| §2 爱情 | "我们最近沟通不太顺" | 需要区分：是吵架频率高了还是冷战不说话了？前者是冲突风格问题，后者是连接断裂问题——处置方式完全不同。 |
| §3 友情 | "朋友不多但都是很久的" | 不需要鼓励他"多社交"。他的社交需求本身不高，质量 > 数量。不要在他说"不想去聚会"时给"你应该多出去走走"的建议。 |

#### 翻译层的工程实现

翻译表不是一次写死的，而是随着使用迭代：
- **初始翻译**：冷启动阶段从 yourself-skill 产出的 persona 文件提取 + 用户在对话中直接描述的自我认知
- **持续修正**：用户每次说"不对，我其实是..."时，触发纠正流程更新对应翻译
- **冲突发现**：当 AI 观察到的行为模式和用户自述标签矛盾时，标记 `[⚠️ 待确认]`，下次对话试探性提出

> **存储**：翻译表的**真理源**是 `_system/behavior_rules.md`（Markdown，人工可读可改），SQLite 仅做热查询缓存。这是旁路索引中唯一不可由 `rebuild-index` 自动重建的例外——因为规则是用户通过 `/correct` 人工精修的，不是从其他地方派生的。rebuild 时从 Markdown 恢复。

### 3.2 非功能需求

- **隐私**:语料默认本地;高敏感分类可强制走本地模型;无遥测。
- **性能**:常用问答 P95 < 5s;冷启动语音转写 < 3s。
- **可用性**:7×24 在线;离线时 Telegram 端排队,在线后回灌。
- **可演进性**:数据格式必须是纯 Markdown + YAML frontmatter,任何工具都能接管。
- **可移植性**:所有数据脱离系统仍可读(不被 Khoj 锁定)。
- **可观测性**:所有 AI 写入操作有 audit log,可回滚。

## 4. 用户交互设计

### 4.1 入口
- **主入口:Telegram Bot**(私人 bot,仅自己使用)。
- **辅助:Obsidian 桌面/移动**(深度阅读、编辑、复盘)。
- **备用:Khoj 自带的 Web UI**(已部署在同一服务器，Telegram 不可用时浏览器直接访问)。
- **可选:iOS 快捷指令**(锁屏一键语音 → Telegram)。

### 4.2 对话模式

#### 模式 A:被动记录(默认)
我说一句 → AI 简短共情/确认 → 后台静默归档 → 给我一个一行回执("已记到 §9 情绪 / §1 妈妈")。

#### 模式 B:主动追问
AI 检测到信息缺要素,反问 1–2 个问题(不超过 3 轮)。
例:我说"今天面试挂了" → AI 问"哪家公司,卡在哪一轮?现在心情怎样?"。

#### 模式 C:深度对话
我主动说 `/think` 或"帮我理一下" → 进入决策/疏导流程 → AI 调用对应模板。

#### 模式 D:冷启动蒸馏
我说 `/onboarding` → AI 引导我提供原材料（微信导出/日记/口述）→ 一次性蒸馏出 Self Memory + 行为翻译表初稿 → 我审阅修正。详见 §7。

#### 模式 E:复盘

### 5.1.1 存储设计原则:Markdown 主存 + 旁路索引

> 这是回应"AI 生成结构化数据,还该用 Obsidian 吗"的核心决策。

**结论:用 Obsidian,但加一层 SQLite 旁路索引**。

| 层 | 作用 | 谁来写 | 是否真理 | rebuild 行为 |
|---|---|---|---|---|
| Markdown 文件 | 给人看、给人改、长期保存 | AI(主) + 人(随时接管) | ✅ 单一真相源 | 不参与 rebuild |
| YAML Frontmatter | 关键结构化字段(name/age/score/status...) | AI 写入,人可改 | ✅ 跟随 Markdown | 不参与 rebuild |
| Qdrant 向量库 | 语义检索 | 后端从 Markdown 派生 | ❌ 可重建 | 从 Markdown 重建 |
| SQLite 关系索引 | AI 做聚合/统计/精确查询 | 后端从 frontmatter 与正文标签派生 | ❌ 可重建 | 从 Markdown 重建 |
| SQLite 行为翻译表 | 标签→行为规则的热缓存（§2.4） | 冷启动蒸馏 + 用户 `/correct` 迭代 | ❌ **不可重建** | Markdown 真理源在 `_system/behavior_rules.md`，SQLite 仅做缓存 |

> **行为翻译表是唯一例外**：翻译规则由用户通过 `/correct` 人工迭代精修，不是从 Markdown 自动派生的。因此它的真理源是 `_system/behavior_rules.md`，SQLite 仅做热查询缓存。`rebuild-index` 时翻译表从 Markdown 恢复，不丢规则。

**为什么不直接上数据库?**
- 数据所有权:Markdown 永远可读,任何工具都能接管,Khoj 倒了不影响。
- 编辑友好:Obsidian 移动/桌面端随时手改 AI 写错的地方。
- 可演进:后期换 Open WebUI / 自撸,只需改后端,不动数据。

**为什么不只用 Markdown?**
- 纯语义检索做不到精准聚合(如"过去 30 天和妈妈冲突频率""所有 sensitivity=high 的决策")。
- 字段级查询需要结构化索引,但这种索引必须是**派生的**,不能是真理。

**索引派生流程**:
- 文件保存 → 监听器(fswatch)触发 → 解析 frontmatter + 正文 → 写入 SQLite + Qdrant。
- 可任意时刻 `rebuild-index` 重建,不丢任何信息。

#### 5.3.1.1 证据强度分级（所有 AI 写入内容必填）

> 借鉴 yourself-skill 的"有依据才写"原则，并进一步建模为四级证据强度。这直接影响 AI 在后续对话中使用该信息的置信度和措辞。

| 等级 | 含义 | 来源 | AI 引用规则 |
|------|------|------|-----------|
| **L1 确证** | 用户直接陈述或确认过 | 用户说"我认为/我觉得/我决定"、用户在 `/correct` 中确认 | 可直接引用，用肯定语气 |
| **L2 推断** | AI 从 ≥2 次行为模式中归纳 | 多次情绪日记中出现相同模式、多次决策呈现相同倾向 | 引用时标注"从过去 N 次类似情况观察到"，用"倾向于/通常"语气 |
| **L3 猜测** | AI 单次推断，未经确认 | 单次对话中观察到的可能模式 | 用试探性语言（"我注意到...有没有可能..."），等待用户确认或否定 |
| **L0 已否定** | 用户明确纠正过的内容 | 用户说"不对"、`/correct` 否定了某条 | **永不使用**，并记录在 Correction 日志中作为反面案例 |

> **写入规则**：AI 每次写入时必须同时评估 `evidence` 等级。L2/L3 的内容在写入时标记，L3 的内容在下次对话中应试探确认。当 L3 被用户确认后升级为 L1，被否定后降级为 L0。

人物档案（§1-§4 的"关于他人的部分"）仍保持独立文件，Self Memory 只记录**我在关系中的模式和态度**，不重复记录他人的具体信息。

**Self Memory 与 §3.4 行为翻译表的关系**：
- Self Memory 是**叙述性的自我认知**（给人读的）："我重视成长多于稳定，但过去三次重大决策都选了稳定"
- 行为翻译表是**规则性的 AI 中间表示**（给 AI 执行的）："当用户谈论职业选择时，不要快速倾向'应该冒险'的建议——他的实际行为显示他优先安全"
- 两者从同一组原材料（冷启动蒸馏）产生，但存放在不同层：Self Memory → Markdown，行为翻译表 → SQLite

#### 5.4.2 纠正流程（Correction 闭环）

> 借鉴 yourself-skill 的 correction_handler 机制。回滚是倒退，纠正是进化。用户纠正后，AI 不只是撤销写入，而是**更新对用户的理解**。

```
用户说 "不对" / "我不是这样" / `/correct`
  │
  ▼
[识别纠正类型]
  ├─ 事实纠正 → Self Memory（"我没在那上过学"）
  ├─ 性格纠正 → 行为翻译表（"我生气不会沉默，我会直接说出来"）
  ├─ 分类纠正 → 归档规则（"这不属于情绪日记，这是灵感"）
  │
  ▼
[确认理解] AI 回述：我理解你是说{旧认知}不对，应该是{新认知}，对吗？
  │
  ▼
[执行纠正]
  ├─ 修改对应文件的被纠正内容 + 标注 [已纠正,见 Correction #N]
  ├─ 追加 Correction 记录到 _system/corrections.md
  ├─ 如果是行为翻译表的纠正 → 更新 SQLite 翻译规则
  │
  ▼
[立即生效] 下一条回复就应体现新认知
```

**Correction 记录格式**：
```markdown
### Correction #3 — 2026-06-09
- 层级：行为翻译表 / §9 情绪-生气
- 原文：生气时先冷静分析 → 然后用理性方式表达不满
- 纠正为：生气时先沉默 10-20 分钟 → 然后直接说出不满（有时带情绪，但说完就过去了）
- 用户原话："我生气的时候不会那么理性，我通常会先不说话，等一会再直接说出来，有时候语气还是会冲"
- 影响：后续所有涉及"用户生气"场景的行为预测都应按新规则
```

**冲突本身就是信号**：当新旧信息矛盾时（如旧认知"成长 > 稳定" vs 新行为连续三次选择稳定），应标记 `[⚠️ 冲突]` 并反馈给用户。这类冲突是 §20 模式识别的核心输入——它揭示用户自我认知和实际行为之间的差距。

#### 5.4.3 `/undo` 实现策略

`/undo` 的技术难点：AI 写入后用户可能手动编辑了同一文件，此时简单地"删掉 AI 最后写入的段落"可能误伤用户手改。

**策略：写入位置记录 + 修改检测**

```
AI 每次 write_note 时，在 audit.log 记录：
  {
    "id": "write-2026-06-09-1830-001",
    "file": "05_journal/emotion/2026-06-09.md",
    "action": "append",           # append | insert | create
    "location": {
      "line_start": 24,           # 写入起始行号
      "line_end": 48,             # 写入结束行号
      "byte_offset": 1024,        # 写入前文件大小（用于检测是否被后续修改）
      "content_hash": "sha256..." # 写入内容的 hash
    },
    "timestamp": "2026-06-09T18:30:00+08:00"
  }
```

`/undo` 执行流程：

```
1. 查找 audit.log 中最近一条写入记录（write_id）
2. 检查目标文件：
   a. 文件大小 == byte_offset + 写入内容长度 + 后续 AI 写入? → 无用户修改，安全回滚
   b. 文件行号 range 的内容 hash 匹配原始 content_hash? → 内容未被改动，安全回滚
   c. 上述均不匹配 → 文件在 AI 写入后被用户手动修改过
3. 情况 c 的处理：
   → 提示用户："这个文件在你上次写入后又被编辑过，我无法安全地自动回滚。
      你想 (A) 我打开文件让你手动删除 / (B) 跳过这次 /undo / (C) 强制回滚（可能丢失你手改的内容）"
```

**限制**：
- `/undo` 只回滚最近一次 AI 写入，不支持跨多次写入的回滚
- 如果用户手动编辑了文件后不做 `/undo`，audit.log 不感知（不需要——下次 AI 读文件自然看到最新版本）
- `/undo` 回滚后在 audit.log 记录回滚事件，不删除原始写入记录（保留完整审计链）

```
[规整] → 统一为 messages.jsonl (sender, receiver, time, text, attachments)
   ↓
[隐私分级] → 按对话方/关键词标记敏感度
   ↓
[摘要切片] → 按对话方 + 时间窗(月/季)分片做 AI 摘要
   ↓
[关联写回 Vault]
   ├─ 06_notes/messages/<人>/<年月>.md  (摘要 + 引用片段)
   └─ 01_people/<人>.md  关系动态追加
   ↓
[向量索引] 原文不入库,仅摘要与关键引用入向量库
```

#### 5.6.3 隐私要求
- 原始 dump 单独目录,默认加密(`_raw_messages_encrypted/`)。
- AI 检索默认只看摘要层,需要原文时显式工具调用 + 二次确认。
- 涉及他人隐私的内容,默认不进任何云模型 prompt;只用本地模型摘要后,再让云模型用摘要工作。
- 提供 `purge_messages(person, before_date)` 一键清理。

### 5.7 多语言处理

| 维度 | 策略 |
|---|---|
| 输入 | 中英混杂自然接受,不做强制语种切换 |
| Embedding | `bge-m3`(多语言,本地)或 `text-embedding-3-large`(云)二选一,统一向量空间 |
| 索引 | 不分语种建库;一个 query 跨语言召回 |
| 改写 | query 阶段双语扩展(原文 + 翻译版本)以提升召回 |
| 生成 | 跟随用户语言;主用中文 |
| 回执 | 简短,用中文 |

### 5.8 三层蒸馏模型（决定"AI 越用越懂我"的关键）

> yourself-skill 只做了一次性蒸馏（原材料 → persona 文件）。但知识库项目的核心价值是**持续进化**——不仅要知道"初始版本的你"，还要能感知"变化的你"。这需要三层蒸馏，而非一次性。

#### 5.8.1 三层结构

```
Layer A：一次性蒸馏（冷启动）          ← yourself-skill 覆盖的部分
  微信/日记/口述 → analyze → Self Memory + 行为翻译表 → L1 基底语料

Layer B：持续蒸馏（每次对话）          ← 知识库项目独有的价值
  每次 Telegram 输入 → 提取行为证据 → 增量更新 Self Memory / 行为翻译表
  → 这是"越用越懂我"的工程实现

Layer C：周期性模式检测（每周/月）      ← 知识库项目独有的价值
  累积 N 条后 → AI 横向对比 → 发现矛盾/新浮现的模式 → 人类确认
  → 例："你说重视健康，但过去 3 个月情绪日记里从未提过运动"
  → 例："你之前说自己'纠结体'，但近一个月 3 次决策都是当天做出"
```

#### 5.8.2 Layer A：一次性蒸馏（冷启动）

| 输入 | 产出 | 存入 |
|------|------|------|
| 微信聊天记录导出（可选） | Self Memory 初稿（5 维度） | `03_self/self_memory.md` |
| 日记/备忘录/笔记 | 行为翻译表初始规则（20 类） | SQLite `behavior_rules` 表 |
| 口述/问卷回答 | 核心人物档案初稿（§1-§4） | `01_people/` 各文件 |

流程同 yourself-skill 的 Step 2-4：导入 → 分析 → 预览 → 审阅修正 → 写入。区别在于：
- 产出是**关于用户的事实理解**，不是模仿用户说话的 persona
- 行为翻译表的翻译方向是"用户是什么样的 → AI 应如何理解他"，而非"用户怎么说话 → AI 就怎么说话"
- 用户对初始产出的修改 = 第一批 Correction 记录，直接纳入闭环

#### 5.8.3 Layer B：持续蒸馏（每次对话）

> 这是 yourself-skill 没有覆盖的部分，也是知识库项目区别于一次性 persona 工具的核心。

每次 Telegram 对话中，AI 在归档写入时**同时判断本条是否包含 Self Memory 增量**：

| 增量类型 | 判断条件 | 动作 |
|----------|---------|------|
| **新事实** | 用户说了之前没记录过的经历/偏好/习惯 | 追加到 Self Memory 对应维度 |
| **新证据** | 已有认知获得了更多行为数据的支持 | 提升 evidence 等级（L3→L2, L2→L1） |
| **新矛盾** | 行为与已有认知不一致 | 标记 `[⚠️ 冲突]`，加入冲突队列，在复盘时提出 |
| **无增量** | 本条不涉及自我认知 | 仅归档到对应分类，不动 Self Memory |

**注意**：持续蒸馏只更新 Self Memory 和旁路的翻译表，**不直接修改 L1 基底语料**。L1 基底语料在月度复盘时由人工从 Self Memory 精炼后更新（见 Layer C）。

#### 5.8.4 Layer C：周期性模式检测

| 周期 | 触发 | 做什么 |
|------|------|--------|
| **每周** | 周日自动 | 拉取本周情绪日记 + 决策 + 灵感 → 生成周复盘草稿 → 推送"要不要做周复盘？" |
| **每月** | 月末自动 | 横向对比本月数据：情绪趋势、决策对照价值观、冲突队列回顾 → 生成月度 Self Memory 更新建议 → 人工确认后更新 L1 基底语料 |
| **每季** | 季末建议 | 模式识别：反复出现的行为模式、自我认知 vs 实际行为的系统性偏差 → 更新 §20 模式档案 → 人工确认 |

#### 5.8.5 对话时的语境注入

```
┌─────────────────────────────────────────────┐
│            每次对话注入 AI 的语境             │
├──────────┬──────────────────────────────────┤
│ L0 骨架  │ §6.1 system prompt + Persona     │ <2k token │
│ L1 基底  │ Self Memory 精炼版                │ 4-8k token │
│          │ （价值观摘要+核心矛盾+关键模式）     │            │
│ L2 召回  │ RAG 从 Vault 检索相关文件          │ 2-4k token │
│          │ （情绪日记/决策档案/人物档案等）     │            │
│ L3 翻译  │ 行为翻译表热点条目                  │ <1k token  │
│          │ （与当前话题相关的翻译规则）         │            │
└──────────┴──────────────────────────────────┘
```

> **L1 基底语料维护规则**：每次月度复盘时由 AI 提议、人工确认更新。不接受 AI 自动写入——基底语料的漂移会系统性影响 AI 的所有判断。

## 6. AI Agent 设计

### 6.1 主 Agent System Prompt（分层结构）

```
你是 X 的私人 AI（代号"隐"），长期陪伴他、了解他、帮他记录与思考。

============================================================
Layer 0：硬规则（不可被任何其他指令覆盖）
============================================================

0.1 你是一个了解 X 的陪伴者，不是 X 的克隆体。用你自己的温和中性风格说话，
    不要模仿 X 的口吻、口头禅、表达风格。
0.2 不突然变得完美、温柔、无条件包容。保持适度的克制和中立。
0.3 永远先共情、再分析、最后才建议。非他主动要建议时不给建议。
0.4 不评判、不灾难化。心理疏导场景遵循"事件-感受-想法-行为"框架。
0.5 可以表达不确定。"我不确定，但从过去的记录来看..."
0.6 不要变成"人生导师"模式。你的角色是陪练和镜子，不是权威。

============================================================
基底认知（L1 永久注入，每次对话一定在场）
============================================================

你已经"内化"了关于 X 的以下信息（详见注入的 self_memory 精炼版）：
- 核心价值观与核心矛盾（如"渴望被认可又讨厌暴露脆弱"）
- 反复出现的行为模式与防御机制
- 人生时间线摘要与关键转折点
- 核心人物（父母/伴侣/密友 ≤5 人）的关键档案摘要

这些不是检索来的，是你已知的。回答时自然融入，而非生硬引用。

============================================================
证据使用规则
============================================================

你掌握的关于 X 的信息有不同的可信度，使用时必须区分：

| 证据等级 | 引用方式 |
|----------|---------|
| L1 确证（他直接说过）           | 肯定语气。如："你之前提过..." |
| L2 推断（多次观察到）           | 标注来源。如："从过去 3 次类似情况来看，你倾向于..." |
| L3 猜测（单次观察，未确认）     | 试探语气。如："我注意到一个可能的规律...你觉得有道理吗？" |
| L0 已否定                       | 永不使用。 |

如果 L3 被你确认，下次可作为 L1 使用。如果被你否定，立即遗忘并记录。

============================================================
语言信号解读
============================================================

你的目标不是模仿 X 的说话方式，而是**理解他的语言信号背后的状态**：

- 他说"算了" → 在特定语境下可能意味着失望而非真的放弃 → 追问语境，不要假设
- 他说"还行" → 配合历史 pattern 可能是回避深入讨论 → 简短确认后试探是否想展开
- 深夜长段落 vs 白天短句 → 可能反映不同的情绪状态 → 调整回应深度
- 他突然用很正式的语气 → 可能这个话题对他来说敏感 → 降低追问压力

============================================================
操作规则
============================================================

1. 默认用模式 A（静默归档）。只在信息缺要素或他选择 /think 时启动追问。
   追问每次不超过 2 轮。

2. 写入前确认目标文件路径与片段。有冲突时优先在 inbox 暂存。
   写入时必须评估并填写 evidence 等级。

3. 新旧信息矛盾时，标记 [⚠️ 冲突] 而非覆盖。在回执中提醒用户有需要确认的矛盾。

4. 决策场景（/think）：必须调用 §14 模板，必须显式对照 §10 价值观和已有的价值观-行为冲突记录。

5. 引用 Vault 内容时必须给出 [[文件路径]]，严禁编造。

6. 高敏感（sensitivity≥high）的语料，在内部推理后只在最终输出中保留必要信息。

7. 行为翻译表（L3 注入）中与当前话题相关的规则，优先于通用推理。

============================================================
工具
============================================================

- classify(text) → tags[]
- read_note(path)
- write_note(path, section, content, evidence_level)
- search_rag(query, k)              # 调用 Khoj API
- query_behavior_rules(tags) → rules[]   # 查行为翻译表
- schedule_followup(when, prompt)
- switch_persona(name)
- record_correction(type, old, new, user_quote)
```

### 6.2 Persona 列表

| Persona | 触发 | 风格 |
|---|---|---|
| 默认"隐" | 默认 | 中性、克制、温和 |
| 倾听者 | 心情低落/`/persona listener` | 只共情、镜映、不分析 |
| 教练 | `/think`、决策类 | 苏格拉底式提问、逼近本质 |
| 分析师 | 技术/职业类 | 结构化、列表化、给数据 |
| 损友 | `/persona buddy` | 调侃、不正经、降压 |

### 6.3 多模型路由策略

> 全部走云端 API,不引入本地模型。隐私边界靠**信任厂商 + 内容脱敏**而非本地化。

#### 6.3.1 模型清单
| 模型 | 角色 | 强项 | 用在哪 |
|---|---|---|---|
| Claude Sonnet/Opus | 主大脑 | 共情、长上下文、推理稳 | 默认全部场景 |
| DeepSeek-V3/R1 | 备份 + 降本 | 中文好、价格低 | 大批量摘要(§18 通讯历史)、备份 |
| `text-embedding-3-large` 或 `bge-m3` | 嵌入 | 多语言 | 索引(§6.7) |

#### 6.3.2 路由规则
| 场景 | 路由 |
|---|---|
| 默认对话(F1–F5) | Claude |
| 心理疏导(模式 C) | Claude(共情质量优先) |
| 决策辅助(F6) | Claude |
| 大批量摘要(F17 通讯导入) | DeepSeek(降本) |
| 文章吸收(F11) | DeepSeek 一档摘要 → Claude 二档关联 |
| Claude 限流 / 故障 | 自动降级到 DeepSeek |
| 用户显式 `/model deepseek` | 切到 DeepSeek |

#### 6.3.3 内容脱敏(进 prompt 前)
- 真实姓名 → 代号(妈妈/伴侣/A/B...)。
- 手机号、身份证、地址、银行卡号 → 占位符。
- 脱敏映射表只存本地,云端模型只见代号。
- 这一层与厂商无关,默认对所有云模型生效。

## 7. 冷启动方案（蒸馏式）

> **核心转变**：v0.2/v0.3 的 7 天访谈大纲依赖用户的自我觉察能力——你说自己是什么样的人，AI 就接受。但人对自己常有盲区。蒸馏式冷启动从真实的聊天记录和日记中反推你是谁，比自我陈述更客观。访谈作为蒸馏的**补充**而非替代——用来填补蒸馏无法覆盖的"内心戏"部分。

### 7.1 冷启动流程（约 2-3 小时集中 + 后续轻量审阅）

```
Phase 1：原材料准备（用户自行，约 30 分钟）
  ├─ 微信聊天记录导出（如有）：PC 端备份 → WechatExporter / PyWxDump → 导出 txt
  ├─ 日记/备忘录/笔记：已有就提供，没有跳过
  ├─ 对自己的一句话描述（口头禅、决策方式、生气时的反应等，想到什么写什么）
  └─ 核心人物一句话（父母/伴侣/密友，谁？关系怎样？）

Phase 2：AI 蒸馏分析（自动化，约 5-10 分钟）
  ├─ 运行 self_analyzer：从原材料提取价值观、习惯、记忆、人际关系、成长轨迹
  ├─ 运行 persona_analyzer：提取说话风格、情感模式、决策模式、人际行为
  └─ 生成行为翻译表初始规则

Phase 3：用户审阅修正（交互式，约 1-2 小时）
  ├─ AI 展示 Self Memory 摘要（5 维度，每维度 3-5 条要点）
  ├─ AI 展示行为翻译表摘要（关键标签→规则，约 10-15 条）
  ├─ 用户逐条确认/修正/补充
  ├─ AI 根据修正生成 Correction 记录
  └─ 产出：self_memory.md v0 + 行为翻译表 v0 + 核心人物档案 v0

Phase 4：补充访谈（可选，约 30-60 分钟）
  ├─ AI 识别蒸馏的盲区（如"从聊天记录看不出你的内心感受"）
  ├─ 针对性提问填补空白
  └─ 产出：self_memory.md v1
```

### 7.2 蒸馏 vs 访谈的互补关系

| 维度 | 蒸馏擅长 | 访谈擅长 |
|------|---------|---------|
| 行为模式 | ✅ 从聊天记录中有大量真实数据 | ❌ 自我描述往往和实际行为有差距 |
| 说话风格 | ✅ 口头禅、语气词、标点习惯 | ❌ 人很难准确描述自己的语言习惯 |
| 决策方式 | ✅ 从多次选择中归纳 | ❌ 人容易美化自己的决策过程 |
| 内心感受 | ❌ 聊天记录可能只展示表层 | ✅ 直接问"那时候你心里怎么想的" |
| 价值观 | ⚠️ 从行为反推，需确认 | ✅ 直接问，但可能和实际行为矛盾 |
| 童年/创伤 | ❌ 很少有人会在聊天里详细描述 | ✅ 结构化的人生时间线访谈 |

### 7.3 原材料不足的退路

如果用户没有微信导出或日记（很常见），退路顺序：

1. **有备忘录/碎片记录** → 用这些做蒸馏（即使数据量少，能抓到部分 pattern）
2. **完全无材料** → 用 yourself-skill 的 Step 1 基础信息 + 口述（§7.1 Phase 4 的访谈作为主方案而非补充）
3. **两者都不要** → M-1 关卡 B（Claude Projects 手工对话 2 周），在对话中自然积累初始语料

无论哪种退路，最终产出件都是一样的：`self_memory.md` + 行为翻译表 + 核心人物档案。

### 7.4 冷启动完成的判定标准

- [ ] Self Memory 5 个维度都至少有 2 条内容（"原材料不足"标记也算完成，但需显式标注）
- [ ] 行为翻译表至少覆盖了用户明确提到过的 5 个以上标签/自我描述
- [ ] 核心人物（父母/伴侣/≤2 位密友）的人物档案有基础信息 + 1 条以上关系动态
- [ ] 用户读完产出后说"是的，这是我"（同 §0.1 关卡 A 判定标准）

## 9. 成功指标

- **使用频率**:6 个月后日均交互 ≥3 次。
- **覆盖度**:6 个月后 20 类中至少 15 类有内容,§9 日均 1 条。
- **召回质量**:在主观评估中,AI 回答能"引用我的真实过去"≥80%。
- **决策辅助**:至少 3 次重要决策走过 §14 模板,事后复盘评价"有帮助"。
- **心理价值**:主观感受"被理解感"显著强于通用 ChatGPT。
- **数据可控**:全程零云端泄露事件;任意时刻能一键导出。

## 10. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|

| 数据泄露 | 极高 | 见 §8 隐私安全 |

| Khoj 上游断更 | 中 | 数据是 Markdown,可平迁到 Open WebUI / 自撸 |

## 12. 待确认 / 开放问题

### 已决策项

1. **微信导出工具** → **M3 阶段再选型**。MVP 不涉及,但建议优先调查 PyWxDump(支持 macOS,2026 年仍活跃)。
2. **Obsidian 移动端同步** → **iCloud 优先**(macOS + iOS 原生支持,零配置)。如 iCloud 延迟不可接受,换 Syncthing。决策理由:iCloud 对 Markdown 小文件同步成熟稳定,且 Obsidian 官方推荐。
3. **加密备份** → **git-crypt + 私有 GitHub**。理由:(1) 仓库已有每日 git 提交流程,加 git-crypt 零额外运维;(2) GitHub 私有仓免费;(3) 不需要引入额外工具。git-crypt 密钥离线存储(1Password / 物理 USB)。
4. **Embedding 模型** → **MVP 走 Khoj 默认配置**,不自行决策。Khoj 默认使用 `sentence-transformers` 本地模型,无需 GPU。M2 根据检索质量决定是否切换到 `text-embedding-3-large`(云)或 `bge-m3`(本地)。
5. **脱敏映射** → **M2 引入**。MVP 期 Anthropic API 默认不训练用户数据,且用户量=1,风险可控。敏感内容用户手动改代号后输入。
6. **DeepSeek 数据策略** → **M3 接入前确认**。M1/M2 全部使用 Claude,不引入 DeepSeek,降低安全评估复杂度。M3 大批量摘要场景再评估。
7. **L1 基底语料 token 预算** → **M2 实测后定**。初始分配 8k token,Self Memory 5 维度精炼版 6k + 核心人物摘要(≤5 人) 2k。超出的部分做分层裁剪:优先保留核心价值观和核心矛盾,生活习惯可降级到 RAG 检索。
8. **关卡 A 不通过的 plan B** → 按 §0.1 处置:不通过则项目不进入 M1。不要提前准备 plan B——关卡 A 的设计目的就是**低成本证伪**,如果证伪了说明这个方向不值得投入,强行换方案是沉没成本谬误。
9. **Khoj 二次开发深度** → **M1 仅通过 HTTP API 调用**。理由:(1) 解耦,随时可换后端;(2) M1 的归档逻辑直接写文件到 Vault,不需要嵌入 Khoj;(3) 如果 M2 出现性能瓶颈或需要深度定制,Khoj 是 Python 项目,届时再评估 fork/嵌入。M1 阶段只调用 Khoj 的 `/api/chat` 和 `/api/search` 两个端点。

### 仍待确认(需外部信息)

- **DeepSeek 训练数据条款**(M3 前确认):如条款不允许不训练,则只用 Claude。
- **微信导出工具实测**(M3 前确认):在 macOS 15 + 最新微信版本上测试 PyWxDump,确认可用性和导出完整性。

### 13.3 核心组件规格

#### 13.3.1 Telegram Bot (`src/bot/`)

```
入口: POST /webhook (Telegram → FastAPI)
模式: Webhook(生产) / Polling(本地开发)
处理链:
  消息到达 → middleware.py 预处理
           → commands.py 判断是否命令
           → 是命令 → 分发到对应 handler
           → 否 → 进入 orchestrator.agent 处理
```

**关键实现细节**:
- 语音消息:Telegram 下载 `.ogg` → Whisper API 转写 → 原文 + 转写文本一并传入 agent
- 图片消息:Telegram 下载 → 可选 OCR(Apple Vision API 本地,M3) → 归档到 `_attachments/`
- 命令实现见 §4.3 命令清单,每个命令一个 async function
- conversation handler:利用 `python-telegram-bot` 的 `ConversationHandler`,为 `/think` 和 `/onboarding` 维护多轮对话状态

#### 13.3.2 主 Agent (`src/orchestrator/agent.py`)

```
职责:接收归一化后的用户输入 → 调用 Claude API(tool calling) → 返回回执

核心函数:
  async def process_input(
      text: str,
      user_id: str,
      mode: Literal["default", "think", "onboarding", "recap"]
  ) -> AgentResponse:
      # 1. 构建 system prompt (§6.1 分层结构)
      # 2. 注入上下文(L1 Self Memory + L2 RAG 召回)
      # 3. 调用 Claude API,注册 tools
      # 4. 处理 tool use 循环
      # 5. 组装面向用户的回执
      # 6. 返回

Tools 注册给 Claude:
  - classify(text) → tags[]
  - read_note(path, section?) → content
  - write_note(path, section, content, evidence_level)
  - link(from_path, to_path)
  - search_rag(query, k=10) → results[]
  - schedule_followup(when, prompt)
```

#### 13.3.3 分类器 (`src/orchestrator/classifier.py`)

```
输入: raw_text + 可选的对话上下文
输出: [{category_id, confidence, rationale}]

策略:
  M1: 直接由主 Agent 的 classify tool 完成
       (Claude 对 20 类 + 中英混合分类效果足够,
        不引入额外分类模型)
  M2+: 高频类别(§9 情绪日记)考虑微调小分类器降延迟/成本,
        但保留 Claude 分类做兜底

回退: 置信度 < 0.7 → 归入 00_inbox,回执注明"已暂存收件箱"
```

#### 13.3.4 Vault 写入器 (`src/vault/writer.py`)

```
核心责任:
  - write_note(): 追加/插入/创建 Markdown 文件
  - 自动生成 YAML frontmatter(模板填充)
  - 写入前做冲突检查(新旧信息矛盾?)
  - 写入后在 audit.log 记录位置和 hash(供 /undo 使用)
  - 写入后触发索引增量更新(通知 index/watcher.py)

关键约束:
  - 绝不覆盖人工写入的内容(通过行号检测,见 §5.4.3)
  - 绝不删除文件
  - 高敏感内容(sensitivity≥high)只写入,不在日志中出现原文
  - write_note 是唯一能修改 Vault 的函数——所有路径都收敛到这里
```

### 13.5 核心流程 — 伪代码

#### 13.5.1 被动记录流程 (模式 A)

```python
# src/orchestrator/agent.py

async def handle_passive_record(text: str, user_id: str) -> str:
    # 1. 构建 system prompt(L0 硬规则 + L1 Self Memory 摘要)
    system = build_system_prompt(persona="default")

    # 2. RAG 检索相关上下文(L2)
    rag_results = await khoj.search(text, k=5,
        filters={"sensitivity": ["low", "mid"]})  # 不召回高敏感

    # 3. 查询行为翻译表(L3,与当前话题相关的规则)
    behavior_rules = await query_behavior_rules(text)

    # 4. 调用 Claude
    response = await claude.chat(
        system=system,
        messages=[{"role": "user", "content": text}],
        context={"rag": rag_results, "rules": behavior_rules},
        tools=TOOLS,
        tool_choice="auto"
    )

    # 5. 处理 tool use 循环
    while response.has_tool_use():
        for tool_call in response.tool_uses:
            if tool_call.name == "classify":
                tags = await classifier.classify(tool_call.input["text"])
                # 追加结果到消息
            elif tool_call.name == "write_note":
                await vault_writer.write(
                    **tool_call.input,
                    audit_id = generate_audit_id()
                )
            elif tool_call.name == "schedule_followup":
                await followup_store.add(**tool_call.input)

        response = await claude.continue_with(tool_results)

    # 6. 返回回执给用户
    return response.final_text  # 如:"已记到 §9 情绪 / §1 妈妈"
```

#### 13.5.2 检索问答流程

```python
async def handle_search(query: str) -> str:
    # 1. Query 改写:中英双语扩展
    expanded = expand_query(query)  # "焦虑" → "anxiety 焦虑 anxious"

    # 2. 向量召回
    results = await khoj.search(expanded, k=20)

    # 3. 重排:BM25 + recency + frontmatter 过滤
    reranked = rerank(results, query,
        recency_weight=0.3,
        bm25_weight=0.7
    )

    # 4. 精确过滤:查 SQLite 做字段级筛选(可选)
    if need_exact_filter(query):
        filtered = await sqlite.filter(
            type=["emotion_log"],
            created_after="2026-05-01",
            tags_contain=["§9"]
        )
        reranked = intersect(reranked, filtered)

    # 5. LLM 生成:强制引用文件路径
    answer = await claude.generate(
        system="基于提供的 Vault 文件回答。引用格式:[[路径]]",
        context=reranked[:5],  # top 5 文件全文
        question=query
    )
    return answer
```

#### 13.5.3 `/undo` 流程

```python
async def handle_undo(user_id: str) -> str:
    # 1. 查 audit_log 最近一条写入
    last_write = await audit_log.get_latest(user_id, action="write")
    if not last_write:
        return "没有可回滚的操作。"

    # 2. 检查目标文件当前状态
    file_path = last_write.file_path
    current_content = await vault_reader.read_raw(file_path)
    detail = json.loads(last_write.detail)

    # 3. 三种情况判断
    if current_content[detail["line_start"]:detail["line_end"]] == \
       detail["original_content"]:
        # 情况 a/b:内容未被改动,安全回滚
        await vault_writer.remove_lines(
            file_path,
            detail["line_start"],
            detail["line_end"]
        )
        await audit_log.append(Undo(...))
        return f"已回滚 {file_path} 的最近一次写入。"

    else:
        # 情况 c:文件被人工修改过
        return (
            f"⚠️ {file_path} 在你上次写入后被手动编辑过,"
            f"无法安全自动回滚。\n"
            f"(A) 打开文件手动删除 / (B) 跳过 / (C) 强制回滚?"
        )
```

## 16. 优化策略

### 16.1 Prompt 优化

#### 16.1.1 分层注入 + 缓存

```
每次对话的 prompt 结构(目标:<15k token):
┌────────────────────────────────────────┐
│ L0 硬规则 + Persona           ~2k     │ ← 可缓存(System message)
│ L1 Self Memory 精炼版          ~6k     │ ← 可缓存(仅在月度更新时变)
│ 核心人物摘要(≤5人)            ~2k     │ ← 可缓存(仅在人物更新时变)
│────────────────────────────────────────│
│ L2 RAG 检索结果(本次相关)     ~3k     │ ← 每次不同,不可缓存
│ L3 行为翻译规则(本次相关)     ~1k     │ ← 每次不同,不可缓存
│────────────────────────────────────────│
│ 当前对话历史                  ~3k     │ ← 每次不同,不可缓存
│ 总计                          ~17k    │
└────────────────────────────────────────┘
```

**Anthropic Prompt Caching 策略**:
- L0+L1 部分(10k tokens)标记为 cache breakpoint → 后续请求 cache hit → 成本降低 90%
- 缓存 TTL 5 分钟,高频对话期间有效
- 缓存失效后自动重建,不丢信息

#### 16.1.2 System Prompt 精简原则

- 每条规则 ≤2 句;能删的先删再发布
- 不使用"你可能应该""或许可以考虑"等软话占位符——只在需要软化时用
- 用表格/列表而非段落(LLM 解析列表更准确)
- 每 10 次对话后检查一次实际 prompt 长度,超过 20k token 立即裁剪

### 16.2 成本优化

#### 16.2.1 M1 阶段成本估算

| 操作 | 模型 | 平均 token/次 | 预估日次数 | 日成本(USD) |
|------|------|-------------|-----------|------------|
| 被动记录(模式 A) | Claude Sonnet | 输入 12k / 输出 200 | 10 | ~$0.36 |
| /search 检索问答 | Claude Sonnet | 输入 15k / 输出 500 | 3 | ~$0.14 |

| **日均合计** | | | | **~$0.58** |

| RAG 检索 | Khoj Qdrant 预热;常用查询结果 LRU 缓存(5min TTL) | /search <3s |
| 索引更新 | fswatch 延迟合并(5s 内多次变更合并为一次重建) | 消除高频写入的索引抖动 |
| 冷启动 | L0+L1 prompt 预编译,服务启动时加载到内存 | 首问延迟降低 30% |

### 16.4 Storage 优化

#### Vault 增长管理

- **预期增长率**:日均 5-10 个新文件(情绪日记 ×1 + 灵感碎片 ×2 + 对话归档 ×2-5)
- **年增长**:约 2000-3600 个 Markdown 文件,总计 50-100 MB
- **无需特殊优化**:这个量级下 SQLite + Qdrant 都能在 1s 内完成查询

#### 索引重建策略

```
触发条件:
  - 主动执行 /rebuild-index(极少)
  - Khoj 升级导致向量库 schema 变更
  - 索引文件损坏

执行流程:
  1. 锁定写入(Telegram 消息排队)
  2. 清空 SQLite + Qdrant(备份到 /tmp)
  3. 遍历 Vault 所有 .md 文件 → 解析 frontmatter → 写入 SQLite
  4. 调用 Khoj API 全量重新 embed → 写入 Qdrant
  5. 从 _system/behavior_rules.md 恢复行为翻译表(唯一不可派生的数据)
  6. 解锁写入,处理排队消息

预计耗时:100 MB Vault → ~5 分钟(本机,含 embed)
```

## 附录 D:API 契约草案

### D.1 内部组件契约

#### D.1.1 VaultWriter

```python
@dataclass
class WriteNoteInput:
    path: str                    # 相对 Vault 根路径,"05_journal/emotion/2026-06-09.md"
    section: str                 # "## 事件" | None=追加到末尾
    content: str                 # Markdown 正文
    evidence_level: Literal["L1","L2","L3","L0"]
    sensitivity: Literal["low","mid","high","top"]
    tags: list[str]              # ["§9","§1-mom","conflict"]
    linked: list[str] | None     # [["mom"]]
    source: str                  # "telegram_text"|"telegram_voice"|"obsidian"|"web"

@dataclass
class WriteNoteOutput:
    file_path: str               # 实际写入的完整路径
    audit_id: str                # write-2026-06-09-1830-001
    line_start: int
    line_end: int
    content_hash: str            # sha256
    conflict_warnings: list[str] # 如有冲突,警告列表
```
